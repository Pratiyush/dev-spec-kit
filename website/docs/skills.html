Compose, don't vendor

dev-spec-kit is the enforcement + traceability layer; it deliberately does not duplicate generic dev craft. Skills compose with superpowers (MIT) — brainstorming, systematic debugging, git-worktree discipline, parallel subagent dispatch — and pick up where superpowers stops: proofs, gates, and the graph. Install both alongside each other.

dev-spec-kit-workflow

skills/dev-spec-kit-workflow/SKILL.md
Auto-fires in dev-spec-kit projects

The main event. Drives spec-driven development with dev-spec-kit — route the request, write EARS specs with @check bindings, derive evidence-bound tasks, TDD, prove checks with real test runs, build the Verified Traceability Graph, record approvals, and generate graph-derived PR bodies.

Fires when

  • The user asks to build or change code in a project with .dev-spec-kit/ present
  • The user mentions "dev-spec-kit", "specs", "EARS", or "traceability"

Hard rules (RFC-2119)

  • MUST NOT mark, claim, or imply a task is complete while any bound check is failing or unrun.
  • MUST NOT weaken, skip, or delete a failing test to make it pass; fix the code or escalate.
  • MUST show real command output for check runs — never paraphrase a result that did not happen.
  • Commits MUST be authored by the human.
  • MUST confirm with the user before proceeding when mode.confirmFirst: true.
  • Quick mode still writes at least one test — no exceptions.

The loop

Route → Spec → dev-spec-kit spec tasks → TDD red → Prove green → dev-spec-kit task done → dev-spec-kit graph build → dev-spec-kit approve → dev-spec-kit pr

dev-spec-kit-intake

skills/dev-spec-kit-intake/SKILL.md
Brain-dumps · tickets · raw ideas

Turn a raw idea, voice-dump, or ticket into a dev-spec-kit intake file and route it. Raw ideas live as files under .dev-spec-kit/intake/ — versioned, re-routable, never lost.

Fires when

  • The user brain-dumps a request without structure
  • The user pastes a Jira, GitHub, or GitLab ticket
  • The user points at an idea that has no spec yet

Intake file shape

Files go in .dev-spec-kit/intake/<slug>.md with YAML frontmatter: id, source (raw|github#123|gitlab#45|jira:ABC-7), date. The user's words are captured verbatim first — messy is fine; the original is never lost.

Process rules

  • Capture verbatim first, then refine.
  • Run dev-spec-kit route --file and report the mode + reason.
  • If vague: ask clarifying questions, one at a time, ≤5, each with a recommended answer.
  • Branch by mode: research → report only, no code; quick → create task directly; full-spec → hand off to dev-spec-kit-spec-author.
  • Write-back to ticket only if intake.writeBack allows.

dev-spec-kit-spec-author

skills/dev-spec-kit-spec-author/SKILL.md
Writing or revising specs

Write high-quality dev-spec-kit specs — EARS acceptance criteria with @check bindings in .dev-spec-kit/specs/*.md. The spec is the source of truth: dev-spec-kit spec tasks derives evidence-bound tasks from it, and every criterion's @check binding becomes a proof obligation in the graph.

Fires when

  • Authoring or revising a spec
  • Converting a ticket or idea into requirements
  • Criteria are vague, untestable, or unbound

Quality checks before handoff

  • Every criterion observable from outside?
  • Every criterion bound to ≥1 @check?
  • No compound sentences (split them)?
  • Unhappy paths covered or explicitly deferred with a note?
  • IDs stable and sequential?

Ref syntax by runner

  • Maven: ClassName#methodName
  • vitest/jest: path/to/file.test.ts::test name
  • pytest: path/to/test_file.py::test_name

dev-spec-kit-review

skills/dev-spec-kit-review/SKILL.md
Before any PR · after graph build is green

Two-pass adversarial code review for dev-spec-kit changes. Verification proves the criteria; review hunts what criteria can't see. Run after dev-spec-kit graph build is green — review is not a substitute for proofs, and proofs are not a substitute for review.

Fires when

  • The user asks for a review or adversarial review
  • Before any PR (dev-spec-kit graph build green first)

Pass 1 — blind (diff only)

Read ONLY the diff (git diff main...HEAD) in a fresh subagent with no spec or prior conversation. Hunt: logic errors, edge cases, security smells, concurrency, resource leaks, error-handling gaps. The implementer finished suspiciously confidently — do NOT take their word for anything.

Pass 2 — full context

Load specs, graph, and tasks. Hunt: criteria satisfied in letter but not intent; scope creep; spec gaps the implementation revealed; reuse violations (code-graph query); silent failures (swallowed exceptions, default fallbacks that mask errors); behavioral coverage vs line coverage.

Output rules

  • Findings are a list — never auto-fix (config review.fixFindings: "list").
  • Each finding: severity (BLOCKER/MAJOR/MINOR/NOTE) · location · what · why · suggested fix.
  • Zero findings is a legitimate outcome. Manufacturing nitpicks is review fatigue, not quality.
  • The reviewer MUST NOT approve their own implementation.

dev-spec-kit-retro

skills/dev-spec-kit-retro/SKILL.md
After features · after failures · on /retro

Run the dev-spec-kit learning loop — capture lessons after a feature or incident into .dev-spec-kit/learnings.md, propose promotions into the laws, and turn fixed bugs into permanent regression checks.

Fires when

  • After completing a feature (default: per-feature from config)
  • After any surprising failure or incident
  • When the user says "retro"

Lesson lifecycle

  • Append with trigger, lesson text, confidence (evidence-based: low/medium/high), scope (project/global).
  • Propose a promotion for every entry — law, permanent regression test, or config change.
  • Human must approve. Confidence is not approval.
  • A fixed bug → permanent regression test bound via @check. A bug that can come back silently was never fixed.
  • Recurrence in 2+ projects → scope: global.

Before new work

Scan learnings.md for OPEN entries and warn when the current task pattern-matches a known mistake (the repeat-warning mechanic).

dev-spec-kit-finish

skills/dev-spec-kit-finish/SKILL.md
finish · land · merge · wrap up

The completion ritual — run when a feature's tasks are done and it's time to land the branch. Entry is EARNED, the menu is FIXED, destruction is TYPED, and every step lands in the journal.

Fires when

  • The user says "finish", "land", "merge", or "wrap up"
  • After dev-spec-kit pr is ready and the human wants to close the loop

Entry gate (fresh evidence only)

  1. Run dev-spec-kit graph build — MUST exit 0. Old green output does not count.
  2. Run dev-spec-kit status — all tasks for this feature DONE.
  3. Run dev-spec-kit pr --title "..." — gate verdict green.

If any step fails: STOP. Report what blocked. Do not present the menu.

The fixed menu (exactly 4 options)

  1. Merge locally — merge to the base branch, verify the suite on the merged result.
  2. Push + PR — push and open the PR with the generated body (dev-spec-kit pr --create).
  3. Keep as-is — leave the branch; record why in the journal.
  4. Discard — destructive. The human MUST type discard exactly. Anything else aborts.

No fifth option. No commentary or recommendations. Wait for the choice.

Provenance-checked cleanup

On Discard: delete branch; remove the worktree ONLY if it lives under .worktrees/ or .claude/worktrees/. Never clean up a workspace dev-spec-kit did not create.
