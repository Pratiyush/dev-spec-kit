Concepts

Core concepts

The ideas behind dev-spec-kit's architecture — why each piece exists and how they fit together.

The Verified Traceability Graph

The VTG is dev-spec-kit's central data structure. It overlays specification, proof, and code edges on top of graphify's code-knowledge graph (Java, TypeScript, Python, and 20+ more languages). The result is a single structure that answers: "is every requirement proven, by which check, at which tree-hash, touching which code nodes?"

Node types and proven edges in the VTG
📋 Requirement → 📐 Criterion → ✅ Check (proof) → 🏗️ Code node → 📦 PR

The graph is written to .dev-spec-kit/graph.json by dev-spec-kit graph build. Every edge carries metadata: proof state, tree-hash, timestamp, and actor. The guard hook reads this graph before any PR creation.

The graph is append-only in principle — evidence accrues. The board command renders the graph as human-readable boards; the trace command prints the requirement→criterion truth table; affected traverses the code-graph reverse edges.

Proof states

Every @check binding has a proof state. The done-gate and PR-gate both use worst-of rollup — one RED binding makes the whole task/criterion red, regardless of how many green bindings exist.

GREEN
A real test run passed at this tree-hash. The tree is not dirty. This is the only state that satisfies the done gate.
RED
A real test run failed. Fix the code or the test (never weaken a test to pass it). Re-run to record a new proof.
STALE
Code changed after this proof was recorded. The tree-hash no longer matches. Run dev-spec-kit drift to re-prove.
UNRUN
No proof has been recorded yet for this binding. Treated identically to RED by the gate — unproven is blocked.

Tree-hash identity

The key insight behind dev-spec-kit's proof model: proofs stamp the tested tree, not the commit SHA. This solves two subtle problems:

  1. Dirty-tree greens: if a proof stamped HEAD while the tree was dirty, it was vouching for uncommitted code. dev-spec-kit stamps the tree-hash + a dirty flag, so a dirty-tree run's status is tracked accurately.
  2. Commit invalidation: if a proof stamped the commit SHA, committing the tested code would move HEAD and instantly invalidate every proof. By comparing tree-hashes, dev-spec-kit detects drift (tree changed) vs stability (tree unchanged, different commit) correctly.

EARS acceptance criteria

Every acceptance criterion in a dev-spec-kit spec is written in EARS syntax (Easy Approach to Requirements Syntax). EARS sentences are machine-parseable and force observable, testable behavior.

# EARS patterns
 
WHEN <event> THEN the system SHALL <observable response>
IF <condition> THEN the system SHALL <response>
WHILE <state> the system SHALL <continuous behaviour>
WHERE <feature set> the system SHALL <capability>
 
# Good: observable, testable
WHEN a session has been idle for >= 30 min THEN the system SHALL return HTTP 401.
 
# Bad: implementation detail, not a behaviour
The system SHALL use a HashMap to store sessions.
 
# Bad: vague, untestable
The system SHALL handle sessions gracefully.

Rules for EARS criteria in dev-spec-kit:

  • One behaviour per criterion. Split compound sentences.
  • The SHALL clause must be observable from outside — a response, a state change, or an emitted event.
  • Never an implementation detail ("uses a HashMap") or a vibe ("is fast", "handles gracefully").
  • Every criterion must carry ≥1 @check binding.
  • Criterion IDs (REQUIREMENT_<AREA>-NN) are stable forever — never renumber; deprecate instead.

Laws

.dev-spec-kit/laws.md is the agent's constitution — the rules it must always obey, regardless of the request. Laws are checked before any action. Conflicts between a request and a law trigger a refusal or warning (configurable via rules.onConflict).

dev-spec-kit's own hard rules (from .dev-spec-kit/laws.md):

  • Commits are authored by the human, never co-authored by the AI.
  • A task is not "done" until its bound checks pass (evidence-bound completion).
  • Reuse existing code before writing new; follow the surrounding patterns.
  • Every feature gets an adversarial review pass before PR — verification and review catch different bug classes; neither substitutes for the other.

Laws promote from learnings: when a lesson is confirmed and approved, it can be promoted into laws.md as a permanent rule. Personal laws can be inherited and overridden per-project.

Gate packs

Gate packs (GATE-PACKS-01) are user-editable config data that define additional requirements for specific kinds of changes. A pack specifies:

  • sections — spec sections that must be present (e.g. "Security", "API Contract")
  • kinds — check kinds required (e.g. api for the contracts pack)
  • triggers — keywords that floor the routing mode to full-spec when found in the request
Menu yes, mandate no Gate packs are a menu — none are required by default (gates.require: []). Ceremony stays proportional to change size. Add packs to gates.require only for changes where the extra rigor earns its keep.

The built-in security pack triggers on: auth, login, password, token, secret, payment, crypt, session, permission, sql. Any request containing these keywords is floored to full-spec regardless of the routing classification.

Investigative gates (GATE-FACTS-01)

When gates.facts: "on", the guard shifts to DENY→FORCE→ALLOW mode. The first edit to a gated file is blocked until the agent gathers named facts — importers, schemas, verbatim instructions. The investigation creates context that demonstrably changes output quality. The gate is opt-in, time-boxed to 30 minutes, and bounded at 500 entries.

The learning loop

.dev-spec-kit/learnings.md is an append-only lesson ledger. After each feature (or surprising failure), lessons are added with a trigger, lesson text, confidence level, and scope. Unpromoted lessons are OPEN; they must be PROMOTED (into laws) or HARDENED (into permanent regression tests) to count.

A lesson only counts once it is PROMOTED or HARDENED. Logged-but-unpromoted lessons recur — that is the failure mode this loop exists to kill.

Lesson lifecycle:

  1. Append to learnings.md after a feature or incident.
  2. Propose a promotion: law, permanent regression test, or config change.
  3. Human approves. Confidence is evidence, not feeling — and confidence is not approval.
  4. A fixed bug becomes a permanent regression test bound via @check. A bug that can come back silently was never fixed.
  5. A lesson recurs in 2+ projects → scope elevates to global; move to personal default rules.

Boards: LEDGER + TRACKING

dev-spec-kit board regenerates two board files from the journal and graph:

  • LEDGER.md — the complete traceability ledger: every requirement, criterion, proof state, and evidence link.
  • TRACKING.md — task-level progress board with traffic-light proof states, assigned tasks, and wave groupings.

These boards are derived — they cannot lie. Unlike manually edited markdown, their content is always consistent with the proof record. Run dev-spec-kit board whenever you want a fresh view.

The input contract

dev-spec-kit's architecture separates three things:

  • Engine code — changes only when dev-spec-kit itself evolves. Lives in src/.
  • Skills — dev-spec-kit's voice and workflow definitions, versioned under skills/.
  • Project inputs — the only thing that changes per project. Lives in .dev-spec-kit/.

The tool never changes for a new stack or project — only the inputs do. This means:

  • A new test stack? Add an entry to verify.runners in config — no code change.
  • A new law? Append to laws.md — no code change.
  • A new lesson? Append to learnings.md — no code change.
  • A new routing rule? Add a gate pack to gates.packs — no code change.
