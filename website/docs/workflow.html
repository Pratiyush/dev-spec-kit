Workflow

The Rivet workflow

Verification by construction — every stage gates the next. You cannot merge unproven work.

Rule zero A claim of "done" is worthless; only an executed check counts. The CLI enforces this — rivet task done physically refuses without green proofs.

1. Route — classify the request

Every change starts at the front door. rivet route classifies a raw request into one of three modes, applies gate-pack security triggers, and (when mode.confirmFirst is true) waits for your confirmation before proceeding.

$ rivet route "add OAuth2 login"
Mode: full-spec
⚠ Security trigger matched: login — mode floored to full-spec
Reason: auth keyword in request + security gate pack active
Confirm? [y/n]:

Routing modes:

  • research — investigate and report only. No code changes in this mode.
  • quick — one small change, one test. No full spec pipeline.
  • full-spec — EARS spec → tasks → TDD → proof → PR. For new behaviour.

Override with -m full-spec. Route a file from the intake folder with --file .rivet/intake/idea.md.

2. Spec — EARS criteria with @check bindings

Write (or have Rivet draft) a spec file under .rivet/specs/. The spec is the source of truth — everything downstream is derived from it.

# Feature: OAuth2 login
 
> User story: As a user, I want to log in with GitHub OAuth2.
 
## Requirement REQUIREMENT_AUTH-01 — OAuth callback
 
WHEN the OAuth provider returns a valid code
THEN the system SHALL exchange it for a token and create a session.
 
@check kind=integration ref=OAuthIT#callbackExchangesToken
 
## Requirement REQUIREMENT_AUTH-02 — invalid code
 
IF the OAuth code is invalid or expired
THEN the system SHALL return HTTP 400 and NOT create a session.
 
@check kind=unit ref=OAuthServiceTest#rejectsInvalidCode

EARS syntax: WHEN/IF/WHILE/WHERE … the system SHALL …. One behaviour per criterion. Split compound sentences. Every criterion needs ≥1 @check binding — unbound criteria are flagged UNVERIFIED and never become tasks.

3. Tasks — spec-derived, evidence-bound

Run rivet spec tasks. It reads every @check binding in every spec file and creates (or syncs) tasks. The command is idempotent — re-run it after adding new criteria. Existing proofs are never clobbered.

$ rivet spec tasks
✓ T4 REQUIREMENT_AUTH-01 — OAuth callback [bound: OAuthIT#callbackExchangesToken]
✓ T5 REQUIREMENT_AUTH-02 — invalid code [bound: OAuthServiceTest#rejectsInvalidCode]
$ rivet task start T4
$ rivet status
T4 IN PROGRESS 🔴 OAuthIT#callbackExchangesToken (unrun)
T5 TODO 🔴 OAuthServiceTest#rejectsInvalidCode (unrun)

4. TDD — red phase first

Write the failing test before any production code. The --expect-red flag tells the runner this is a TDD red phase: it skips flaky retries (so it doesn't burn your retry budget) and records the red proof without triggering the done-gate block.

$ rivet check run T4 OAuthIT#callbackExchangesToken --stack java-maven --expect-red
✗ FAILED (expected — TDD red phase) tree: b1d4e02 dirty: true
→ write the production code, then re-run without --expect-red

5. Prove — real runs only

Once the implementation is in place, re-run the checks. The runner invokes the real test framework (Maven, vitest, pytest, etc.) and records the result with a tree-hash and dirty flag. You cannot manually edit a recorded result — only real runs count.

$ rivet check run T4 OAuthIT#callbackExchangesToken --stack java-maven
✓ OAuthIT#callbackExchangesToken PASSED tree: c9e1a77 dirty: false
proof recorded — worst-of rollup: GREEN

Proof states:

  • GREEN — passed at the current tree-hash
  • RED — failed; task blocked until fixed
  • STALE — code changed after the proof; must re-run
  • UNRUN — no proof recorded yet

6. Done gate

rivet task done is the hard gate. It reads the live proof graph and refuses if any bound check is not GREEN. This is not a lint warning — it's an exit-code-1 refusal enforced by the CLI.

$ rivet task done T4
✓ T4 done 🎉 (1/1 checks green)
$ rivet task done T5
✗ T5 BLOCKED — bound checks not all green:
🔴 OAuthServiceTest#rejectsInvalidCode — last run: UNRUN
Run: rivet check run T5 OAuthServiceTest#rejectsInvalidCode --stack java-maven

7. Graph build

Fuse the spec, the journal, and the graphify code graph into .rivet/graph.json. Exit 1 if any proof is red or stale. The Verified Traceability Graph overlays requirement→criterion→check→proof edges on top of graphify's code-knowledge graph.

$ rivet graph build
Refreshing graphify index…
Fusing specs + journal + code graph…
✓ graph green (2 requirements · 2 criteria · 2 proofs · all GREEN)
Written: .rivet/graph.json

Use --no-refresh to skip the graphify re-index when the code graph is known to be current.

8. Drift detection

Proofs carry the tested tree-hash. If code changes after a green run, that proof goes stale. rivet drift finds stale proofs and re-runs their checks in one move.

$ rivet drift
⚠ 1 stale proof found:
OAuthIT#callbackExchangesToken — proved at c9e1a77, current tree: f2b0d34
Re-running stale checks…
✓ OAuthIT#callbackExchangesToken PASSED tree: f2b0d34 dirty: false
✓ All proofs current.

Use --dry-run to list stale proofs without re-running. Use --stack to provide a fallback stack for proofs recorded before stacks were journaled.

9. Human approval

Evidence-bound done ≠ reviewed. Record the human gate separately — approval is on top of verification, never a substitute. Tasks must be DONE before they can be approved.

$ rivet approve T4 T5 --note "OAuth flow reviewed; timing and error paths correct"
🛡️ approval recorded → .rivet/approvals/T4-T5-2026-06-12.json

10. PR

rivet pr generates .rivet/pr-body.md from the graph: a full traceability table of requirements → criteria → proofs, with worst-of rollup coverage numbers. The guard hook blocks gh pr create while proofs are not green.

$ rivet pr --title "feat: OAuth2 login" --create
Generating PR body from graph…
✓ guard pr — 2/2 requirements proven GREEN
Creating PR via gh…
https://github.com/org/repo/pull/43
The hook blocks PR creation A PreToolUse guard hook runs rivet guard pr before any gh pr create invocation. Missing graph, stale proofs, or red checks all block — you cannot work around this by calling gh directly. Use rivet unlock (with a time-boxed human approval) if you need to override.

After the PR is open, run the rivet-finish skill to land the branch cleanly.
