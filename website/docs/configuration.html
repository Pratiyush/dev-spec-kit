Reference

Configuration reference

Every knob in .dev-spec-kit/config.json, its default value, and what it controls. All defaults are sourced directly from src/config/schema.ts (zod schema).

How config works dev-spec-kit ships with opinionated defaults — TDD on, gates on, even quick-mode writes a test. Every knob is overridable in .dev-spec-kit/config.json. The schema version is 1; dev-spec-kit applies defaults for any omitted field.
Minimal config An empty object {} is valid — every field has a default. You only need to set what you want to change.

project

Basic project identity and stack detection

project.name
"untitled"
Human-readable project name used in board headers and journal metadata.
project.platforms
[]
Codebase platforms (descriptive) — an ARRAY: multiple languages per project is supported and expected (e.g. ["typescript", "electron", "python"]). NOT runner stacks — runner ids like node-vitest belong to check run --stack / verify.runners. Drives stack inference and init --platforms best-practice packs. (Replaces project.stacks; the legacy key is ignored harmlessly.)
Allowed: "java-maven" | "java-gradle" | "spring" | "quarkus" | "node" | "typescript" | "electron" | "react" | "next" | "angular" | "python"

spec

Idea → acceptance criteria → spec

spec.style
"both"
Spec authoring style.
checklist | stories | both
spec.acceptanceCriteria
"tool-drafts"
Who authors acceptance criteria.
tool-drafts — dev-spec-kit writes a draft for human review; user-writes — human writes criteria directly
spec.criteriaFormat
"gherkin"
Criteria syntax. Gherkin (the default) = Scenario: blocks with Given/When/Then steps; Scenario Outline: + Examples: tables expand one criterion per row (one @check binds them all). EARS = WHEN/IF … THEN the system SHALL …. Both always parse and bind — off-format criteria lint (warn-only). Every requirement also needs ≥1 negative/failure criterion (gates.negativeFloor, on by default).
gherkin | ears | plain | mixed
spec.breakdownDepth
"feature-story-task-subtask"
Decomposition hierarchy depth.
feature-story-task-subtask | task-subtask
spec.estimates
true
Generate effort estimates for tasks.
spec.autoDependencies
true
Automatically detect and record task dependencies from spec analysis.
spec.diagram
true
Generate a dependency/flow diagram alongside the spec.
spec.gapHunting
"propose"
Hunt for missing edge cases and error-handling, then propose extra tasks.
off | propose — list gaps for approval; auto — add tasks automatically
spec.riskWarn
true
Warn and suggest splitting big or risky changes before starting implementation.
spec.livingPlan
"update-ask"
Whether the plan can update mid-build.
frozen — plan locked after approval; update-ask — propose updates for approval; update-auto — update automatically
spec.onVague
"clarify"
What to do when the request is vague.
clarify — ask questions first; guess-flag — proceed with assumptions flagged

verify

What "done" means — the moat, made configurable

verify.kinds
["unit","integration","api","e2e"]
Which proof kinds a criterion may bind to.
unit | integration | api | e2e | visual | parity
verify.coverage
null
Coverage gate as a percentage. null = judge by criteria coverage, not a number.
verify.blockDoneOnFail
true
The moat. A task cannot be marked done while bound checks fail. This is the enforced invariant; disabling it removes the hard gate.
verify.everyCriterionNeedsCheck
true
Every acceptance criterion must bind to ≥1 executed check. Unbound criteria are flagged UNVERIFIED by dev-spec-kit spec tasks.
verify.runApp
false
Spin up the application for api/e2e checks. Requires verify.app.start to be set.
verify.ui
"off"
Visual / UI check method.
off | screenshot | browser | both
verify.sandbox
"local"
Run checks in an isolated sandbox or directly on the machine.
sandbox | local
verify.security
"on-demand"
When security checks run.
off | pre-pr | on-demand
verify.lintTypes
"part-of-done"
Where lint and type checks live.
part-of-done | separate
verify.flaky
"retry-flag"
Flaky test policy.
retry-flag — retry and flag if inconsistent; quarantine — move to quarantine set
verify.runners
{}
Custom check-runner commands, keyed by stack name. Overrides built-in runners when the key matches; unknown keys define new stacks. Each entry: { "cmd": "mvn", "args": ["test", "-Dtest={name}"] }. Placeholders: {"{ref}"}, {"{file}"}, {"{name}"} — use combined --flag={"{name}"} form so nameless refs drop cleanly.
verify.kindRunners
{}
Kind-level runner templates (RUNNERS-01). A binding's kind can carry its own command — e.g. a visual snapshot harness or a parity runner. Same placeholders as verify.runners. Precedence: kindRunners > runners (stack) > builtin.
verify.app.start
[]
argv to start the app for api/e2e checks when verify.runApp is true. Example: ["./mvnw","spring-boot:run"]. Empty = no lifecycle.
verify.app.readyUrl
null
URL polled until it responds before checks start. null = fixed 1 s grace period.
verify.app.readyTimeoutMs
30000
Maximum milliseconds to wait for the app's ready URL. Integer ≥ 1. Default: 30 seconds.

true
Enable wave parallelism. When true, dev-spec-kit wave plan/start can dispatch tasks concurrently.

Max concurrent worktree tasks per wave. Default 6 — avoids rate-limit wipeouts. Integer ≥ 1.

"worktree"
Isolation strategy for parallel tasks.

parallel.coordinator
true
Run a coordinator agent that monitors wave progress and handles file-clash serialization.

gates

Gate packs and investigative edit gates (GATE-PACKS-01 + GATE-FACTS-01)

gates.facts
"off"
Opt-in investigative edit gate (DENY→FORCE→ALLOW). When "on", the guard demands that the agent gather named facts (importers, schemas, verbatim instructions) before the first edit on a gated file. The investigation creates context that changes the output. Window: 30 min; bounded at 500 entries.
off | on
gates.require
[]
Pack names enforced on every spec. Empty by default — ceremony stays proportional. List any of the pack keys defined in gates.packs.
gates.packs
(see below)
User-editable gate pack definitions. Each pack: { sections, kinds, triggers }. sections: spec sections required; kinds: check kinds required; triggers: keywords that floor the routing mode to full-spec. Built-in packs:
  • security: sections ["Security"], triggers: auth/login/password/token/secret/payment/crypt/session/permission/sql
  • contracts: sections ["API Contract"], kinds ["api"]
  • nfr: sections ["NFR"]
  • rollback: sections ["Rollback"]
