How dev-spec-kit works

Under the hood: the journal, how a proof is recorded and aged, what the gate actually checks, and how to extend the tool without touching its code.

Three sources of truth, one graph

dev-spec-kit holds no opinions in memory. Every command re-derives the world from three durable sources and fuses them into the Verified Traceability Graph:

specs/*.md
requirements + EARS + @check × journal.jsonl
executed check results × graphify-out/graph.json
code nodes + imports → graph.json
the VTG

The builder (src/engine/graph/build.ts) is a pure function over those three inputs. There is no hidden database — delete graph.json and dev-spec-kit graph build recreates it byte-for-byte from the journal, specs, and code graph. That is what makes the boards "unable to lie": they are derivations, not assertions.

The journal: append-only truth

.dev-spec-kit/journal.jsonl is an append-only log of typed events — one JSON object per line. Task state, proof state, approvals, and governance events are never stored directly; they are folded from the journal on every read (src/engine/state/journal.ts → tasks.ts). Corrupt or partial lines are skipped, so an interrupted write never bricks the project.

  • You never edit it. Proofs are recorded by real runs; rewriting history is the one thing the tool refuses to let you do quietly.
  • Equal-timestamp ties break toward the worse result — proof state must never depend on iteration order, and when in doubt the gate stays shut.
  • It is committed. The journal travels with the repo, so proofs are reviewable in the PR and reproducible on another machine.

How a proof is recorded

When you run dev-spec-kit check run <taskId> <ref> --stack <stack>, the engine:

  1. Resolves the runner for the check's kind and stack (precedence: verify.kindRunners > verify.runners[stack] > builtin).
  2. Substitutes the ref into the runner's command template and executes the real test process.
  3. Interprets the result — a structured reporter (vitest/jest JSON) when available, otherwise the process exit code. A zero-match selector is a fail, never a silent pass.
  4. Stamps a CheckResult into the journal:
// the proof carries its own identity + diagnostics
interface CheckResult {
  ref: string; // "SessionTest#idle" — the bound selector
  passed: boolean;
  at: string; // ISO timestamp of the run
  sha?: string; // commit at proof time (legacy fallback)
  tree?: string; // content tree-hash of the tested state — the REAL identity
  dirty?: boolean; // were there uncommitted changes?
  stack?: string; // recorded so drift can re-run without asking
  kind?: string; // unit | integration | api | e2e | visual | parity | judge
  flaky?: boolean; // passed only after retries?
  tail?: string; // truncated stdout/stderr on FAILURE
}
A failed proof carries its own diagnosis. On failure the tail captures the last lines of output, so a red edge in the graph tells you why it is red without re-running anything.

Tree-hash identity & staleness

A proof is bound to the content it tested, not the commit it happened on. The builder compares the proof's recorded tree against the working tree's current hash (the .dev-spec-kit/ directory is excluded, so recording a proof never invalidates the proof):

no recorded run → unproven
last run failed → red
passed & tree matches → green
passed & tree differs → stale // code moved since the proof

Each criterion rolls up to its worst proof, and each requirement is green only when every criterion is green. The ranking is fixed in build.ts:

const PROOF_RANK = { unproven: 0, red: 1, stale: 2, green: 3 };
// a criterion with green + stale bindings rolls up to STALE, never green

This is why one weak binding can never be hidden behind nine strong ones — the rollup always reports the strictest state.

The PR gate: one predicate

Every block path — the PreToolUse hook, dev-spec-kit guard pr, and dev-spec-kit pr --create — funnels through a single predicate in src/engine/gate.ts, so there is exactly one definition of "ready". It blocks unless all three hold:

  1. The graph exists and is readable. A missing graph blocks — absence of state is never permission.
  2. Every proven edge is green. Any validates / implements / satisfies edge that is red, stale, or unproven blocks.
  3. The last verify is green at the current tree. A green per-task gate is not a green project; the whole suite must have passed on the code you are shipping (any change since re-arms it).
Exit code 2 = deny. The guard returns exit 2, which Claude Code interprets as "deny this tool call" — so a non-green PR is stopped at the moment gh pr create is attempted, not discovered in review.

The provider contract

The code side of the graph is pluggable behind one file: graphify-out/graph.json. Both providers — bundled revitify and external graphify — emit the same shape, and the builder reads it through loadCodeGraph() (src/engine/graphify/index.ts), tolerant of either vocabulary:

{
  "built_at_commit": "a7f3c91…",
  "nodes": [{ "id": "sym:…", "label": "SessionService", "source_file": "src/…", "community": 3 }],
  "links": [{ "source": "…", "target": "…", "relation": "imports", "confidence": "EXTRACTED" }]
}

revitify is invoked as revitify build <dir> --out graphify-out (or its in-process API); graphify as graphify update .. dev-spec-kit re-indexes only when HEAD has moved since the last index, tracked in .dev-spec-kit/graph-state.json. Because the contract is identical, switching providers is a one-line config change plus a rebuild.

Extend: add a test stack with zero code

dev-spec-kit's engine never changes for a new language — only the inputs do. A new test runner is a config entry under verify.runners, a command template with placeholders the engine substitutes per check:

// .dev-spec-kit/config.json
"verify": {
  "runners": {
    "go-test": {
      "cmd": "go",
      "args": ["test", "-run", "{test}", "./{pkg}"]
    }
  }
}
 
# then bind a criterion to it:
@check kind=unit ref=sessions::TestIdleTimeout
# and prove it:
$ dev-spec-kit check run T1 sessions::TestIdleTimeout --stack go-test

Built-in stacks (java-maven, node-vitest, node-jest, python-pytest) ship with their selector syntax; a config-defined stack uses whatever placeholders its template declares. The seven check kinds (unit · integration · api · e2e · visual · parity · judge) let one criterion demand the right shape of evidence — an API contract test, a visual snapshot, or an LLM judge verdict for the unmeasurable.

The re-prove dance

Because a proof's identity is the working tree, any code or doc change after a green proof makes it stale. The loop back to green is mechanical:

$ dev-spec-kit verify --stamp # re-run the suites, stamp fresh proofs in one pass
$ dev-spec-kit graph build # re-fuse; confirm 0 red / 0 stale
# commit the refreshed .dev-spec-kit/ state

A state-only commit (just .dev-spec-kit/) does not re-stale anything — that directory is excluded from the proof hash — so the dance always terminates. verify --stamp proves every bound criterion from a single suite run, which is far cheaper than per-criterion cold starts once your tests are green.
