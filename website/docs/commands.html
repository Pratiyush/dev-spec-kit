Command reference

Every rivet command with its options and what it does. Sourced directly from src/cli/index.ts.

Every command is safe by default A malformed config.json, missing runner binary, or unavailable stack gives a clean one-line red error message with a meaningful exit code — never a raw stack trace. Every invocation inside a dev-spec-kit project is also automatically recorded in journal.jsonl.
Health
cmd dev-spec-kit doctor

Check prerequisites: Node.js, git, Python, graphify, Java, Maven. Exits 0 only when all required tools are found and the version constraints are met. Run this after dev-spec-kit init and whenever a check runner starts failing unexpectedly.

cmd dev-spec-kit init

Initialize dev-spec-kit in the current git repo. Creates .dev-spec-kit/ with a default config.json, laws.md, empty specs/ and intake/ directories, and an empty journal.jsonl.

-f, --forceOverwrite the existing dev-spec-kit config if present.
Tasks — evidence-bound
cmd dev-spec-kit task create <id> <title>

Create a task bound to one or more check refs. The id (e.g. T1) is stable and permanent — duplicates are refused, not overwritten. Proofs recorded against a task are never clobbered by re-creation.

-c, --check <ref...>One or more check refs this task must prove (e.g. SessionTest#idleTimeout). Space-separated; repeat the flag for multiple.
cmd dev-spec-kit task start <id>

Mark a task in progress. Records the state transition in the journal. While a task is in flight, its bound spec criteria and test files are locked against edits (anti-tamper protection).

gate dev-spec-kit task done <id>

THE GATE. Marks a task done; refuses with exit code 1 unless every bound check has a GREEN proof at the current tree-hash. Not a warning — a hard refusal. Fix the failing checks and re-run; do not argue with the gate.

Config knob: verify.blockDoneOnFail (default: true)

cmd dev-spec-kit status

Print a traffic-light board of all tasks and their proof states, generated from the journal. Use to re-orient after any interruption — the journal is the ground truth.

T1 DONE 🟢 SessionTest#idleTimeout (GREEN)
T2 IN PROGRESS 🔴 ApiTest#createUser (UNRUN)
T3 TODO ⚪ unstarted
Checks — proof recording
cmd dev-spec-kit check run <taskId> <ref>

Execute a check via the stack's real test runner and record the result as a proof in the journal. The proof carries the tested tree-hash and a dirty flag. A RunnerUnavailableError (ENOENT, missing binary) is treated as a tooling error and does not enter the graph as a RED proof — only actual test failures do.

-s, --stack <stack>Required. One of: java-maven · node-vitest · node-jest · python-pytest · any config-defined stack. Precedence: verify.kindRunners > verify.runners (stack) > builtin.
-x, --expect-redTDD red phase. Skip flaky retries for this run. Record the red proof without triggering the done-gate block. Use when writing the failing test first.

Ref syntax by runner:

  • java-maven: ClassName#methodName
  • node-vitest / node-jest: path/to/file.test.ts::test name
  • python-pytest: path/to/test_file.py::test_name
  • config-defined: per verify.runners[stack].args placeholders
Workflow — spec-driven flow
cmd dev-spec-kit route [request]

The front door. Classify a request into research | quick | full-spec, respecting config routing policy and gate-pack security triggers (which floor the mode). Build-intent signals ("I want", "add", "create") veto research keywords.

-m, --mode <mode>Override the classified mode: research | quick | full-spec.
-f, --file <path>Route an external idea file from the intake folder (e.g. .dev-spec-kit/intake/my-idea.md).
cmd dev-spec-kit spec tasks

Create or sync evidence-bound tasks from all @check bindings in .dev-spec-kit/specs/*.md. Idempotent — safe to re-run after editing specs. New @check obligations are diffed in; existing proofs are never overwritten. Done tasks with new obligations are re-opened.

cmd dev-spec-kit approve <taskIds...>

Record the human gate as a signed artifact at .dev-spec-kit/approvals/. Tasks must be DONE. Approval is the human gate on top of verification — never a substitute for proofs.

-n, --note <note>Approval note to include in the artifact.
cmd dev-spec-kit pr

Generate .dev-spec-kit/pr-body.md from the graph: a traceability table with requirement→criterion→proof coverage. Uses worst-of rollup — one red + one green reports partial coverage, never 100%. The PR headline number is always the strictest one.

-t, --title <title>PR title to include in the generated body.
--createAfter generating the body, open the PR via gh. Runs the same guard as the hook — blocked while proofs are not all green.
cmd dev-spec-kit unlock <paths...>

Human escape hatch. Time-boxed unlock of protected files (spec criteria and bound test files that the anti-tamper lock has frozen). Every unlock is journaled as a governance event (🛡️ governance). Default window: 30 minutes.

-m, --minutes <minutes>How long the unlock lasts. Default: 30.
Graph — Verified Traceability Graph
gate dev-spec-kit graph build

Fuse specs + journal + graphify code graph into .dev-spec-kit/graph.json. Exit 1 if any proof is red or stale. The graph must be green before any PR. Gate packs are enforced here (required spec sections, required check kinds).

--no-refreshSkip the graphify re-index even if the code graph is stale. Use when the graph is known to be current.
Queries — read-only

All query commands take a no-write path — they do not rewrite graph.json or trigger re-index as a side effect.

cmd dev-spec-kit trace

Print the requirement→criterion truth table from the graph. Exit 1 while any requirement is unproven. Use before approve/PR to see the full coverage picture.

cmd dev-spec-kit drift

Find red or stale proofs (code changed after the proof was recorded) and re-run their checks in one move. Exit 1 if any remain after re-running.

-s, --stack <stack>Fallback stack for proofs recorded before stacks were journaled.
--dry-runList drifted proofs without re-running. Read-only — does not modify graph.json.
cmd dev-spec-kit affected <label>

Blast radius of a code node — proven edges touching it plus graphify reverse traversal. Useful for impact analysis before changing a shared class or function.

Example: dev-spec-kit affected GreetController

Gates — hook-friendly
block dev-spec-kit guard pr

Block PR creation unless every proof is green. Missing or unreadable graph also blocks — state absence ≠ permission. Exit code 2 signals a block (hook-friendly: Claude Code interprets exit 2 as "deny this tool call"). Called automatically by the PreToolUse hook before any gh pr create invocation.

State & audit
cmd dev-spec-kit log

The audit trail — every recorded action, chronological, with emoji lane. Shows the last 25 events by default.

--jsonEmit raw JSONL instead of the formatted log.
-n <count>Number of events to show. Default: 25. Pass 0 to show all.
$ dev-spec-kit log -n 5
2026-06-12T10:14:03Z 🟢 check.run T1 SessionTest#idleTimeout → GREEN
2026-06-12T10:15:22Z 🔏 task.done T1 (all checks green)
2026-06-12T10:16:01Z 🌐 graph.build green (2 criteria proven)
2026-06-12T10:16:45Z 🛡️ approve T1 "reviewed; looks correct"
2026-06-12T10:17:12Z 📤 pr.create feat/session-timeout → #42
cmd dev-spec-kit resume

Write and print the state-only handoff at .dev-spec-kit/RESUME.md, generated from the journal. Use to hand off context to a new agent session or after an interruption. Pairs with the PreCompact hook that auto-saves a checkpoint at 100% context.

cmd dev-spec-kit board

Regenerate .dev-spec-kit/LEDGER.md and .dev-spec-kit/TRACKING.md from the journal and graph. These are boards that cannot lie — they are derived from the proof record, never manually edited.

Parallel — wave dispatcher
cmd dev-spec-kit wave plan

Group pending tasks into waves: no two tasks in the same wave share bound-check files, capped at parallel.waveSize (default: 6). Outputs the wave grouping for review before dispatch.

cmd dev-spec-kit wave start <ids...>

Create one git worktree per task — fetch-first (branching from origin's current tip, not local HEAD) to avoid stale-branch races. Each worktree runs its task in isolation. Journal writes are file-locked to prevent last-writer-wins collisions across concurrent worktrees.
