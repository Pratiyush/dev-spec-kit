Getting Started

Your first verified change

From zero to an evidence-bound, graph-verified PR in about 10 minutes.

Prerequisites

dev-spec-kit requires three things to be present on your machine. Run dev-spec-kit doctor after installing to confirm all are found.

ToolMinimum versionPurpose
Node.js≥ 22dev-spec-kit CLI runtime
gitany moderntree-hash proofs, worktrees, PR creation
Python≥ 3.10graphify code-graph indexer
graphifylatest20+ language code-knowledge graph

Per-stack build tools (JDK + Maven for Spring/Quarkus, pytest for Python, etc.) are needed for the check runners that match your stacks.

Install

1. Install graphify

The PyPI package is named graphifyy (double y); the CLI command is graphify.

$ pip install graphifyy
$ graphify install
✓ graphify installed

2. Install the dev-spec-kit CLI

dev-spec-kit is built from source (Phase 0 — not yet on npm). Clone the repo and build:

$ git clone https://github.com/Pratiyush/llm-dev-kit.git
$ cd llm-dev-kit
$ pnpm install
$ pnpm build
# Use via: node dist/cli/index.js OR pnpm dev
# Or symlink: npm link (installs `rivet` globally)
Claude Code plugin dev-spec-kit is designed to be driven from Claude Code. Install the plugin (see Skills) and the dev-spec-kit-workflow skill fires automatically whenever you're in a project with .dev-spec-kit/ present.

Initialize your project

Run dev-spec-kit init in any git repo. It creates the .dev-spec-kit/ directory with a default config, empty laws file, specs and intake folders, and an empty journal.

$ cd my-project
$ dev-spec-kit init
✓ .dev-spec-kit/config.json
✓ .dev-spec-kit/laws.md
✓ .dev-spec-kit/specs/ (empty)
✓ .dev-spec-kit/intake/ (empty)
✓ .dev-spec-kit/journal.jsonl
✓ .dev-spec-kit/learnings.md
dev-spec-kit initialized. Run `dev-spec-kit doctor` to verify prerequisites.

Use --force to overwrite an existing config.

Doctor check

Always run dev-spec-kit doctor after init to verify everything is wired correctly.

$ dev-spec-kit doctor
✓ Node.js 22.4.0
✓ git 2.43.0
✓ Python 3.11.7
✓ graphify 1.2.0
✓ .dev-spec-kit/ present
✓ All prerequisites met.

First verified change — end to end

This walkthrough adds an idle-session timeout to a Spring Boot project, proving every step with a real Maven test run.

Step 1 — Route the request

Feed the raw idea to dev-spec-kit's front door. It classifies the work and confirms before proceeding.

$ dev-spec-kit route "add idle-session timeout after 30 minutes"
Mode: full-spec
Reason: new observable behaviour, not a contained fix
Confirm? [y/n]: y

Step 2 — Write the spec

Create .dev-spec-kit/specs/session.md. Every acceptance criterion is an EARS sentence with at least one @check binding.

# Feature: Session management
 
> User story: As a user, I want my session to expire after 30 min idle.
 
## Requirement REQUIREMENT_SESSION-01 — idle timeout
 
WHEN a session has been idle for >= 30 minutes
THEN the system SHALL invalidate the session and return HTTP 401.
 
@check kind=unit ref=SessionTest#idleTimeout
@check kind=integration ref=SessionIT#expiredSessionReturns401
Tip — every criterion needs a check If you can't name the test that would prove a criterion, the criterion isn't done being written. Unbound criteria are flagged UNVERIFIED by dev-spec-kit spec tasks and never become tasks.

Step 3 — Derive evidence-bound tasks

$ dev-spec-kit spec tasks
✓ T1 REQUIREMENT_SESSION-01 — idle timeout [bound: SessionTest#idleTimeout, SessionIT#expiredSessionReturns401]
$ dev-spec-kit task start T1

Step 4 — TDD: write the failing test first

Write SessionTest#idleTimeout before writing any production code. Use --expect-red to record the red phase without wasting retry budget.

$ dev-spec-kit check run T1 SessionTest#idleTimeout --stack java-maven --expect-red
✗ FAILED (expected — TDD red phase) tree: a3b2c11 dirty: true
→ implement the feature, then re-run without --expect-red

Step 5 — Implement, then prove

Write the production code. Once it compiles, re-run the check (without --expect-red) to record the proof.

$ dev-spec-kit check run T1 SessionTest#idleTimeout --stack java-maven
✓ SessionTest#idleTimeout PASSED (tree: a7f3c91 dirty: false)
$ dev-spec-kit check run T1 SessionIT#expiredSessionReturns401 --stack java-maven
✓ SessionIT#expiredSessionReturns401 PASSED (tree: a7f3c91 dirty: false)

Step 6 — Done gate

The gate reads the proof graph. It will block if anything is red or unrun.

$ dev-spec-kit task done T1
✓ T1 done 🎉 (2/2 checks green)

Step 7 — Build the graph, approve, and open the PR

$ dev-spec-kit graph build
✓ graph green (1 requirement · 2 criteria · 2 proofs · all GREEN)
$ dev-spec-kit approve T1 --note "reviewed; timing logic correct"
🛡️ approval recorded → .dev-spec-kit/approvals/T1-2026-06-12.json
$ dev-spec-kit pr --title "feat: idle session timeout" --create
✓ guard pr — all proofs GREEN
https://github.com/org/repo/pull/42
What just happened? The PR body at .dev-spec-kit/pr-body.md contains a full traceability table: every requirement → criterion → check → proof state. Reviewers see exactly what was proved and at which tree-hash.
